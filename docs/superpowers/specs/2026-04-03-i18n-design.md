# Internationalization (i18n) Design Spec

## Overview

Add English/Spanish internationalization to Eloria using `@nuxtjs/i18n`. The system has two translation domains:

1. **Dashboard/marketing UI** — follows the visitor's browser language (auto-detected, cookie-persisted)
2. **Invitation UI** — follows the event's `language` field, chosen by the creator

## Languages

- English (`en`) — default
- Spanish (`es`)
- Easy to extend later; no architectural coupling to just two languages

## Approach

`@nuxtjs/i18n` with `no_prefix` strategy (no `/en/` or `/es/` in URLs), lazy-loaded translation files, browser language auto-detection.

---

## Translation Domains

### Dashboard & Marketing UI

All pages except `/i/[slug]`: navigation, forms, buttons, labels, error messages, dashboard stats, landing page, pricing page.

- Language determined by browser's `Accept-Language` header
- Persisted in a cookie (`i18n_locale`) so it survives page reloads
- Falls back to `en` if browser language is not supported

### Invitation UI

The public invitation page (`/i/[slug]`) and everything rendered on it:

- RSVP buttons and labels ("Joyfully Accept", "Respectfully Decline", etc.)
- Template placeholder strings (extracted from hardcoded HTML)
- AI-generated wording

Language determined by the event's `language` column. On page load, `setLocale(event.language)` overrides the browser default.

### Emails

- **Verification & password reset emails** — follow the browser language at the time of the request (passed as a parameter to the email utility)
- **Invitation delivery emails** — follow the event's `language` field

---

## Database Changes

### Events table

Add column:

```
language TEXT NOT NULL DEFAULT 'en'
```

Stores the invitation language chosen during event creation.

### Templates table

Add column:

```
slug TEXT NOT NULL UNIQUE
```

Used as the namespace key for template-specific translations in the i18n files. The seed script is updated to populate this slug.

---

## Template Placeholder Extraction

Current templates have hardcoded English phrases in their HTML. These are extracted into placeholders and moved to i18n translation files.

### Placeholder mapping (per template, namespaced by slug)

Templates become language-agnostic. Each template defines its own set of placeholder keys. Examples:

**rustic-autumn:**
- `{{familiesLine}}` — "Together with their families"
- `{{honorLine}}` — "We joyfully request the honour of your presence"
- `{{closingLine}}` — "We look forward to celebrating with you"

**modern-minimal:**
- `{{tagLine}}` — "Wedding Invitation"
- `{{venueLabel}}` — "Venue"

**classic-elegant:**
- `{{honorLine}}` — "The honour of your presence is requested"
- `{{closingLine}}` — "We look forward to celebrating with you"

At render time, `TemplatePreview.vue` resolves these placeholders using the i18n translations for the event's language, looking up `templates.{templateSlug}.{key}`.

### Seed templates update

- Add `slug` field to each seed template
- Replace hardcoded English phrases in HTML with `{{placeholder}}` syntax
- Corresponding translations added to `lang/en.json` and `lang/es.json`

## Event Creation Language Selector

Added to **Step 1 (Event Details)** of the creation wizard as a dropdown:

- Options: English, Espanol
- Default: browser's current locale if supported, otherwise `en`
- Saved to `events.language`

The selected language affects:
- Template placeholder preview in Steps 3 & 4
- AI wording generation language in Step 3
- The final public invitation page

---

## AI Wording Generation

Changes to `server/api/ai/generate-wording.post.ts`:

- Accept a `language` parameter in the request body
- Modify the OpenAI system prompt to generate wording in the specified language
- Fallback hardcoded templates (when no API key) have translated versions stored in the i18n files under `ai.fallback.{tone}`

---

## Public Invitation Page

Changes to `pages/i/[slug].vue`:

- API response for `/api/invitations/[slug]` includes the event's `language` field
- On page load, call `setLocale(event.language)` to switch locale
- All UI strings come from i18n `rsvp.*` namespace
- Template placeholders resolved from `templates.{slug}.*` namespace

The entire invitation page renders in the event creator's chosen language.

---

## Email Translations

### Verification & password reset emails

- The email utility receives a `locale` parameter
- Email subject lines and body text come from translation strings
- Locale is determined by the user's current browser language at request time

### Invitation delivery emails

- Use the event's `language` field
- Email subject and body translated accordingly

## Translation File Structure

```
lang/
  en.json
  es.json
```

Top-level namespaces:

```json
{
  "nav": {},
  "auth": {},
  "landing": {},
  "pricing": {},
  "dashboard": {},
  "eventForm": {},
  "eventDetail": {},
  "guests": {},
  "settings": {},
  "rsvp": {},
  "email": {},
  "ai": {
    "fallback": {}
  },
  "templates": {
    "<template-slug>": {}
  },
  "common": {}
}
```

Exact keys are extracted from hardcoded strings in each page during implementation.

## Nuxt Config

```ts
// nuxt.config.ts
modules: ['@nuxtjs/tailwindcss', '@nuxtjs/i18n'],

i18n: {
  locales: [
    { code: 'en', file: 'en.json', name: 'English' },
    { code: 'es', file: 'es.json', name: 'Espanol' },
  ],
  defaultLocale: 'en',
  lazy: true,
  langDir: 'lang/',
  strategy: 'no_prefix',
  detectBrowserLanguage: {
    useCookie: true,
    cookieKey: 'i18n_locale',
    fallbackLocale: 'en',
  },
}
```

## Scope

### In scope

- Install and configure `@nuxtjs/i18n`
- Create `lang/en.json` and `lang/es.json` with all UI strings
- Add `language` column to events table
- Add `slug` column to templates table, update seed script
- Extract hardcoded template phrases into i18n-resolved placeholders
- Language selector in event creation Step 1 (defaults to browser language)
- AI wording generation in event's language (including fallbacks)
- Public invitation page uses event's language
- Email translations (auth emails follow browser language, invitation emails follow event language)
- Marketing pages (landing, pricing) translated

### Out of scope

- Languages beyond en/es
- User-level language preference stored in database
- PDF export translation (follows existing template rendering, so it inherits i18n naturally)
